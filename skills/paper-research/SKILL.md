# /paper-research

**触发条件**：当 agent 的 MEMORY_LOG 中 Dead Ends ≥ 10，或 leader 认为当前方向全面失败、走投无路时，**作为最高优先级的重大事件**触发此 skill。

**目标**：不被现有失败经验困住，主动从学术论文中寻找突破性方法，逻辑严谨地评估其对本项目的适用性。

---

## Phase 1: 检索 (Retrieve)

**目标**：从 MEMORY_LOG 的 Dead Ends 和 Active Problems 中提炼出具体的研究问题，然后系统性地搜索相关论文。

### Step 1.1 — 分析当前困境

读取 MEMORY_LOG.md，提取：
- 所有 Dead Ends（**按时间倒序**，最近的最重要）
- 所有 Active Problems

提炼出 **2-3 个具体、可搜索的研究问题**。格式：
```
Q1: [具体问题描述]
  来源: [哪个 Dead End 暴露了这个问题]
  为什么现有方法失败: [根因分析]

Q2: ...
```

**要求**：
- 问题必须具体，不能泛泛如"深度估计效果差"
- 示例良好问题: "光场深度估计中，angular consistency 如何在非朗伯表面失效？"、"多尺度特征融合在 epipolar geometry 约束不足时如何改进？"
- 示例差问题: "怎么提高 MAE？"、"有没有更好的网络结构？"

### Step 1.2 — 执行搜索

使用 `web_search` 对每个问题进行搜索。

**搜索策略**：
- 每个问题搜索 **3-5 篇论文**
- 优先搜索 **近3年 (2022-2026)** 的论文
- 使用组合关键词，例如: `light field depth estimation angular attention non-Lambertian 2024`
- 每次搜索记录：`论文标题`、`作者/机构`、`发表年份/会议`、`一句话核心方法`

**输出格式**：
```
## 检索结果

### Q1: [问题]
Papers found:
1. [论文标题] — [作者], [年份/会议]
   核心方法: [一句话]
   相关度: [高/中/低] — [理由]

2. ...

### Q2: ...
```

## Phase 2: 验证检索 (Verify)

**目标**：防止 LLM 幻觉，确保找到的论文真实存在且方法描述准确。

### Step 2.1 — 验证论文存在性

对每篇标记"高相关度"的论文，执行：

1. **搜索论文的官方页面**（arXiv, CVPR/ICCV/ECCV 官方页面，或作者主页）
   - 使用 `web_fetch` 抓取论文摘要页
   - 核对标题、作者、年份是否一致
   - 检查论文是否真实存在

2. **验证方法描述的准确性**
   - 读取论文摘要或方法章节
   - 对照 Phase 1 中记录的"核心方法"，检查是否有明显偏差
   - 标记任何**疑似幻觉**的内容（方法描述与摘要矛盾、不存在的引用等）

### Step 2.2 — 交叉验证

- 如果多篇论文引用了同一方法，交叉确认引用链
- 检查该方法是否有**公开代码**（GitHub 链接、paper with code 网站）
- 记录代码可用性（PyTorch/TensorFlow 实现、预训练权重等）

**输出格式**：
```
## 验证结果

### Q1: [问题]
Verified papers:
✓ [论文标题] — [URL] — 验证通过，方法描述与摘要一致
✓ [论文标题] — [URL] — 验证通过，但方法与本项目相关性低于预期
✗ [论文标题] — 验证失败: [原因]（方法描述疑似幻觉，从候选中移除）

Code availability:
- [论文标题]: [有/无] 公开代码 — [PyTorch/TF/其他] — [链接]

### Q2: ...
```

**规则**：任何"高相关度"论文如果无法通过验证，必须从候选中移除。不允许带入 Phase 3。

---

## Phase 3: 评估 (Evaluate)

**目标**：对通过验证的论文进行深度分析，评估其方法对本项目的适用性。

### Step 3.1 — 深度方法分析

对每篇通过验证的论文，深入分析：

1. **方法原理**：用 2-3 句话描述方法的核心技术细节
2. **与现有方法的差异**：本项目中已有的方法 vs 这篇论文的方法，关键区别在哪里
3. **对当前 Dead Ends 的针对性**：这篇论文的方法能否解决 MEMORY_LOG 中记录的某个/某些 Dead Ends
4. **实施难度**：需要多少代码改动、是否需要新的数据预处理、是否有算力要求

### Step 3.2 — 综合排名

将所有通过验证的论文方法按以下维度打分（1-5分）：

| 维度 | 说明 |
|------|------|
| 问题针对性 | 该方法能否解决当前 Dead Ends 中列出的核心问题 |
| 技术可行性 | 在当前代码库/数据/算力约束下实施的可能性 |
| 预期收益 | 实施后可能带来的指标提升 |
| 实现复杂度 | 代码改动量、依赖引入、训练成本 |

**计算加权总分**（问题针对性×2 + 技术可行性 + 预期收益 + 实现复杂度/2），排名前 2-3 的方法进入 Phase 4。

### Step 3.3 — 制定实验路线图

对于 Top 2-3 方法，制定**最小化可行实验**：

```
## 实验路线图

### 实验A: [方法A的名称]
针对的 Dead End: [具体哪个]
针对的 Active Problem: [具体哪个]
核心改动: [1-2句话描述代码改动]
预期结果: [我们期望看到什么指标变化]
风险: [可能失败的原因]

### 实验B: [方法B的名称]
...
```

**输出格式**：
```
## 评估结果

### 论文方法深度分析

#### [论文标题]
原理: [2-3句]
与现有方法的关键差异: [对比]
针对性 Dead End: [哪个]
问题针对性: X/5 | 技术可行性: X/5 | 预期收益: X/5 | 实现复杂度: X/5
综合评分: X.X/10

#### [论文标题]
...

### 综合排名
1. [方法名] (总分 X.X) — 针对: [Dead End 1, Dead End 2]
2. [方法名] (总分 X.X) — 针对: [Dead End 3]
3. ...

### 实验路线图
[见上文 3.3]
```

## Phase 4: 验证 (Validate)

**目标**：在正式进入实验之前，用**逻辑推理 + 最小成本验证**来检验 Phase 3 排名的可靠性。

### Step 4.1 — 逻辑一致性检验

对每个候选方法，回答以下问题：

1. **反事实检验**：如果这个方法真的有效，我们会在训练过程中看到什么迹象？（如：loss 下降更快、val_MAE 稳步下降、r_corr 变稳定等）
2. **互斥性检验**：这个方法和其他候选方法是否互斥？还是可以组合？
3. **历史对比**：该领域之前的 SOTA 方法是什么？与本项目当前基线相比如何？

### Step 4.2 — 最小成本预实验（可选）

如果资源允许，对 Top 1 方法执行**1 epoch 的 dry-run**：
- 复用现有训练框架，仅替换核心模块
- 不保存 checkpoint，仅观察 loss 曲线和初步指标
- 如果 dry-run 明显异常（如 loss = NaN、梯度爆炸），需要重新评估方法

### Step 4.3 — 撰写最终报告

将四个阶段的所有内容整合成一份结构化报告，保存到：
`workspace/paper_research/{YYYY-MM-DD}_report.md`

报告头部必须包含：

```markdown
# Paper Research Report — {日期}

触发原因: [为什么此时触发此 skill]
涉及 Dead Ends: [列出相关 Dead Ends]
涉及 Active Problems: [列出相关 Active Problems]
研究问题: [Phase 1 提炼的问题]

[Phase 1-4 完整内容]

## 重大决策

经过 检索→验证→评估→验证 四环节，
最终推荐执行: [方法名]
推荐理由: [3句话逻辑链]
如果该方法失败，备选: [方法名]

预期突破点: [具体说明这个方法为什么有可能打破现有瓶颈]
风险: [主要风险点]
```

## 重大事件记录要求

每次执行 `/paper-research` 都是一个**重大事件**，必须同时：

1. **更新 MEMORY_LOG.md** — 在 `## Recent Decisions` 中添加：
   ```
   [日期] PAPER_RESEARCH: 触发原因=[原因]。发现[X]篇相关论文，通过[Y]篇验证。推荐方法=[方法名]，评分=[分数]。针对 Dead Ends=[列表]。完整报告: workspace/paper_research/{日期}_report.md
   ```

2. **更新 Active Problems** — 如果找到了新方向但尚未验证，添加新的 Active Problem

3. **写入 Obsidian 日记**（如果启用）— 标记为 `major_research_event`

## 工具清单

| 工具 | 用途 |
|------|------|
| `web_search` | Phase 1 搜索论文 |
| `web_fetch` | Phase 2 验证论文真实性 |
| `read_file` | 读取 MEMORY_LOG、现有代码 |
| `write_to_file` | 保存报告、实验路线图 |
| `run_shell` | 最小成本 dry-run（Phase 4） |

## 约束

- **每次只研究 2-3 个核心研究问题**，不贪多
- **不生成参考文献列表**，只深度分析相关度最高的方法
- **严禁幻觉**：未经验证的论文方法不能进入评估阶段
- **逻辑链必须完整**：每个推荐必须能追溯到具体的 Dead End 和 Active Problem
