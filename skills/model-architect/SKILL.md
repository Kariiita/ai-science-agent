# /model-architect

**触发条件**：
1. Agent 正在设计新模型（THINK phase 中 action=experiment 且涉及新模型设计）
2. 模型训练完成后结果异常（总体MAE恶化或某域严重退化）
3. Agent 需要评估模型架构是否真正实现了其声称的理念（idea validation at code level）

**目标**：确保 Agent 设计的模型在数学、物理、工程层面是合理的，而不是盲目堆砌模块。

---

## Phase 1: 结构扫描 (Structure Scan)

**目标**：快速了解模型的整体架构，计算关键指标。

### Step 1.1 — 读取模型代码

读取模型 `.py` 文件，提取：
- 所有 `nn.Module` 子类及其输入输出形状
- 前向传播 `forward()` 的完整逻辑链路

### Step 1.2 — 计算关键指标

对模型进行以下计算和记录：

```
## 模型指标
| 指标 | 值 |
|------|-----|
| 总参数量 | XXX K |
| 训练数据量 | XXX scenes |
| 参数/数据比 | XXX (params per training sample) |
| GPU 内存 (batch=1, 256x256) | XXX GB |
| 前向传播通道瓶颈 | XXX ch at [位置] |
| 最深卷积层感受野 | XXX px |
```

**关键判断**：
- `参数/数据比 > 5`：严重过拟合风险，模型太复杂
- `参数/数据比 < 0.5`：可能欠拟合，但方向先验足够时不一定是问题
- `GPU > 20GB`：无法在单卡3090上训练

### Step 1.3 — 绘制分支特征流图

用文本ASCII图画出模型的特征流：

```
输入 (B, 81, 3, 256, 256)
│
├── 分支A (XX通道): [描述]
├── 分支B (XX通道): [描述]
│
└── 融合层 → [融合方式] → 输出
```

**关键检查**：每个分支的输出通道数占融合总通道数的比例。如果某分支 < 10%，它的梯度将被其他分支淹没，实际上不会学习。

---

## Phase 2: 物理合理性分析 (Physics Validation)

**目标**：验证模型是否真正实现了其声称的物理原理。

### Step 2.1 — 对比 idea doc 与 code

读取 `idea/idea_report.pdf`（如果存在）中的核心算法描述，逐条对照代码实现：

| idea doc 中的设计 | 代码中如何实现 | 是否对等 |
|-------------------|----------------|----------|
| 角度FFT分解材质 | 代码第X-Y行 | ✅/⚠️/❌ |
| 频带能量分离 (E_low, E_high) | 代码第X-Y行 | ✅/⚠️/❌ |
| RTF秩分析 | ... | ✅/⚠️/❌ |

### Step 2.2 — 检查频域处理正确性

**仅当模型使用 FFT/DCT/频域变换时执行**：

1. FFT 是在哪个维度上做的？（空间？角点？两者？）
2. 是否计算了幅度谱？（✅ 必需）
3. 是否保留了相位谱？（⚠️ 在许多应用中，相位包含关键信息）
4. 是否有分频带分析？（低频 vs 高频能量比 → 物理学上对应漫反射 vs 镜面反射）
5. FFT 结果是否经过 log1p 或 dB 转换？（大幅值会主导梯度）
6. 频域特征与空间域特征如何融合？

### Step 2.3 — 检查多分支贡献均衡性

计算每个分支的**特征贡献比例**和**可学习参数比例**：

```
分支     特征通道数  占比    参数量     占比    权重比
─────    ─────────  ────   ────────   ────   ──────
EPI      320        91%    XXX K      XX%    XX
FFT      32          9%    XXX K      XX%    XX
```

**判断标准**：
- 某分支占比 < 10%：严重欠权重，该分支将被梯度忽略 → **设计缺陷**
- 某分支占比 < 20%：可能欠权重，需验证是否真的有贡献 → **需消融验证**
- 所有分支占比在 20-60% 之间：均衡 → **设计合理**

---

## Phase 3: 数据可行性分析 (Data Feasibility)

**目标**：判断模型复杂度是否与可用数据量匹配。

### Step 3.1 — 统计训练/验证数据

读取 `DATASET_MANIFEST.json`，按域统计：

```
域            训练场景数    验证场景数    数据充足性
───────────   ──────────   ──────────   ──────────
Lambertian       XX           XX         ✅/⚠️/❌
Non-Lambertian   XX           XX         ✅/⚠️/❌
Mixed            XX           XX         ✅/⚠️/❌
```

**判断标准**：
- 训练场景 < 10 且验证场景 < 3：该域数据**不足以支撑有统计意义的学习**
- 验证场景 = 1：MAE 完全不可靠，单场景噪声即等于指标波动

### Step 3.2 — 评估新模型对数据的需求

- 如果新模型增加了大量参数（相比 V3），但没有增加训练数据 → **过拟合风险**
- 如果新模型声称能解决某个域的问题，但该域只有 5 个训练场景 → **不可能**，需要数据增强或合成数据
- 如果新模型引入了频域分支，需要检查 angular resolution (9×9) 是否能提供足够的频率分辨率

---

## Phase 4: 消融实验设计 (Ablation Design)

**目标**：设计系统性的消融实验来验证每个模块的贡献。

### Step 4.1 — 确定消融目标

对模型中的每个分支/模块，问：
1. 这个模块的预期贡献是什么？（如 "FFT分支应改善 Non-Lambertian MAE"）
2. 如何设计一个移除该模块但仍可训练的控制实验？

### Step 4.2 — 设计消融计划

为模型的关键模块设计消融实验：

```
消融实验 1: [模块名]
  控制组: 移除 [模块]，用 [替代方案] 替代
  预期: [指标] 在 [域] 上变化 [方向]，变化量 > [阈值]
  验证的假设: [物理假设]

消融实验 2: [模块名]
  ...
```

### Step 4.3 — 消融优先级

1. **最优先消融**：占比 < 20% 的分支 → 验证它是否真的有用
2. **次优先消融**：新引入的复杂模块 → 验证其必要性
3. **最后消融**：已在 V3 中验证的模块 → 除非怀疑退化

---

## Phase 5: 综合评估与建议 (Synthesis)

### Step 5.1 — 模型质量评分

| 维度 | 评分 (1-5) | 理由 |
|------|-----------|------|
| 物理合理性 | ? | [模型是否实现了其声称的物理原理] |
| 数据匹配度 | ? | [模型复杂度 vs 可用数据量] |
| 计算效率 | ? | [训练时间、GPU内存、收敛速度] |
| 创新性 | ? | [是否提出了 fundamentally new 的设计] |
| 可消融性 | ? | [是否可以系统地验证每个模块的贡献] |

### Step 5.2 — 改进建议

按优先级列出：
- **P0 (必须修复)**：阻断性问题（如某分支无法学习）
- **P1 (强烈建议)**：显著影响性能的问题
- **P2 (锦上添花)**：理论优雅但非紧急的改进

### Step 5.3 — 建议的下一步

- 如果是模型设计阶段 → 给出具体的架构改进建议
- 如果是训练完成后 → 给出消融实验计划
- 如果是结果异常 → 给出诊断和修复方案

---

## Reasoning Principles (MANDATORY)

1. **以物理为本**: 每个模块必须有明确的物理意义，不能盲目堆砌层
2. **数据为尺**: 模型复杂度不能超过可用数据的承载能力
3. **平衡为王**: 多分支模型中，任何占比 < 10% 的分支都值得怀疑
4. **消融为证**: 复杂模型的每个新模块都需要消融实验来证明其必要性
5. **诚实优先**: 如果发现设计缺陷，直接指出，不要为复杂而复杂
